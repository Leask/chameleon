# Chameleon 設計重構草案 v3.0
*(threat-model-first 版本)*

## 0. 文檔目的

這份文檔的目的，不是替 `design_2.md` 做語氣更強烈的包裝，而是
直接把問題重寫。

如果 Chameleon 的目標是面向高壓網路干預環境，從頭構建一個長期
可演進、跨端可落地、性能足夠高、又不把安全建立在脆弱技巧上的
系統，那麼設計必須從威脅模型開始，而不是從傳輸外觀開始。

這一版文檔保留項目的核心設計目標：

- 連通性要穩
- 整體效率要高
- 客戶端與服務端要容易演進
- 多端接入要一致
- 安全邊界要清楚

但同時明確否定幾個危險方向：

- 不再把「像某個熱門協議」當成系統核心價值
- 不再把「隨機擾動」當成主要安全手段
- 不再把「極限帶寬」誤當成性能優化的唯一目標
- 不再把「藏認證」當成完整的安全設計

Chameleon v3 的核心主張只有一句話：

**先定義對手、暴露面、失敗模式與性能指標，再設計傳輸、會話、
控制面與資料面。**

## 1. 對 `design_2.md` 的直接評價

`design_2.md` 已經比最初版本成熟很多，因為它至少接受了幾件事：

- 隨機擾動不是萬能解
- 相變是高風險特徵
- 應用層亂做擁塞控制會把系統搞壞
- 真實靜態掩護比反向代理第三方站點更合理

但它仍然保留了幾個值得立即拆掉的假設。

### 1.1 「指紋完全一致」這個目標不現實

`design_2.md` 提出要讓 QUIC / TLS 指紋與 Chrome 或 iOS
「毫無二致」。這個說法在工程上不成立。

原因不是做不到部分對齊，而是：

- 真實客戶端的表面不是只有 JA3 / JA4
- 指紋是整體行為，不是單一雜湊值
- 同一個瀏覽器版本在不同平台上也不完全一致
- 發包節奏、重傳策略、版本演進、錯誤處理都會暴露差異
- 你能複製一部分表面，但很難長期複製整個生命週期

因此，Chameleon 不應追求「完全一致」，而應追求：

- 不主動暴露自訂特徵
- 不修改可觀測握手語義
- 不引入顯著偏離主流棧的行為
- 在統計上不形成穩定異常群體

也就是說，目標是**控制指紋預算**，不是宣稱完全隱形。

### 1.2 `Noise + 0-RTT + 嚴格重放保護` 不能直接打包成一句話

`design_2.md` 把多個好概念放在一起，但沒有處理它們之間的張力。

如果要有真正嚴格的重放保護，就不能對 0-RTT 抱不切實際的幻想。
0-RTT 可以在部分恢復場景中提高體感，但它不是免費午餐：

- 早期資料天生存在 replay 風險
- 伺服器若設計成接近無狀態，重放判定會更難
- 一旦把敏感語義放進早期資料，安全邊界就會變模糊

因此 v3 的原則很簡單：

- 0-RTT 不是默認能力
- 0-RTT 只用於可重放、可冪等、低風險的恢復場景
- 真正的授權與會話切換必須建立在完整確認之後

### 1.3 「確定性 Padding」仍然可能成為特徵

`design_2.md` 用確定性 padding 取代隨機 padding，思路比亂數擾動好，
但仍然不能直接當成答案。

如果 padding 模式是按會話或 profile 穩定衍生出來的，那它仍然可能
形成可被聚類的規律。尤其當流量用途有限、使用者行為單一、設備類型
固定時，過於規整的 padding 會留下人工設計痕跡。

v3 的態度是：

- padding 不是主體，只能是輔助
- padding 必須有明確成本上限
- padding 只在必要場景啟用
- profile 不能依賴 padding 才成立

換句話說，**不要讓系統的存活性建立在 padding 上。**

### 1.4 「無狀態伺服器只持有根金鑰」是錯誤的運維方向

`design_2.md` 想把服務端做得極輕，這個方向可以理解，但如果線上節點
直接依賴根金鑰衍生會話能力，風險會被放大。

更合理的結構應該是：

- 離線根信任
- 線上簽發或授權層
- 邊緣節點只持有限權短期材料

也就是說，**不能為了部署輕量化，把整個信任鏈壓成單層。**

### 1.5 `HTTP/3 / WebSocket / WebRTC` 可替換，不代表實際成本相同

`design_2.md` 把承載層抽象成可切換介面，這個方向是對的，但說法仍然
過於平面。

這些承載的實際差異非常大：

- 權限模型不同
- 平台支援不同
- 多路復用能力不同
- 中間盒相容性不同
- 電量成本不同
- 連線建立與恢復語義不同

所以 v3 會保留「承載可替換」的抽象，但不再假裝各種承載是等價件。
它們應該被視為不同能力等級的 adapter，而不是只換一個驅動名。

## 2. v3 的設計原則

Chameleon v3 的核心設計原則如下：

### 2.1 先做威脅建模，再做傳輸選型

協議外形不是起點。起點應該是：

- 對手能看到什麼
- 對手能改寫什麼
- 對手能重放什麼
- 對手能封鎖什麼
- 我們能承受哪些失敗
- 我們不能承受哪些暴露

### 2.2 把安全建立在密碼學與狀態機上，不建立在「藏」

任何需要「把東西塞進某個角落」才能成立的認證機制，都不是理想的
系統內核。

v3 要求：

- 身份語義明確
- 授權語義明確
- 會話建立順序明確
- 撤銷與輪換可操作
- 配置更新有信任鏈

### 2.3 把性能建立在 goodput、恢復能力與功耗上

不是只看峰值吞吐，而是看：

- 弱網下的 goodput
- p95 / p99 延遲
- 切網恢復時間
- CPU 消耗
- 行動端耗電
- 長連線穩定性

### 2.4 控制面與資料面分離，且都要最小化

資料面只負責高效、安全、可預測的轉發。

控制面只負責：

- 節點配置
- 信任材料更新
- 策略下發
- 最小化遙測
- 功能灰度與回滾

兩者分離的目的不是形式美觀，而是降低耦合與爆炸半徑。

### 2.5 多端能力先抽象，再實作

先定義：

- 本地接入模型
- 連線生命週期
- 配置模型
- 重試語義
- 日誌與遙測邊界

然後才分別做桌面、行動端與無頭模式實作。

## 3. Threat Model First：先把問題定義清楚

### 3.1 對手能力

v3 假定對手具備以下能力：

#### A. 被動觀察能力

- 觀察完整連線中繼資料
- 提取 TLS / QUIC 可見參數
- 建立流量大小、時間、方向、生命週期特徵
- 將流量與 IP / ASN / 憑證 / 域名做關聯
- 基於大規模樣本做聚類與異常檢測

#### B. 主動干預能力

- 注入丟包、限速、連線抖動
- 發送異常握手或畸形請求
- 進行重放與回放變體測試
- 使用差分方法比對多次回應
- 觸發連線重置、端口封鎖或路徑級降級

#### C. 資產與營運層威脅

- 域名被封鎖
- IP / ASN 被封鎖
- 節點被掃描與長期觀察
- 客戶端配置被截獲
- 線上節點被入侵
- 供應鏈更新被污染

### 3.2 我們真正要保護的東西

v3 需要保護的核心資產不是一句「連得出去」就能概括。
至少包括：

- 通訊內容的機密性
- 身份與授權材料的安全性
- 設備遺失後的可撤銷性
- 伺服器失陷後的歷史內容安全
- 控制面被污染時的可恢復性
- 在干預環境中的可用性

### 3.3 非目標

v3 需要誠實承認一些非目標，否則文檔一定會飄：

- 不保證對全球被動對手的匿名性
- 不保證單一承載永久可用
- 不保證單條連線在任何路徑下都高吞吐
- 不保證單一節點永不被聚焦或封鎖
- 不追求「完美偽裝」這種無法驗證的口號

## 4. 系統不變式

如果一個版本違反以下不變式，就不應進主線。

### 4.1 不修改標準握手的可觀測語義

不要在可觀測握手中加入自訂欄位、奇怪排序、非主流參數組合。
任何偏離都會變成識別面。

### 4.2 授權與會話建立必須是可審計的狀態機

不能依賴「客戶端和服務端心裡都知道」的隱式過程。
每一步都要能回答：

- 這一步在驗證什麼
- 這一步失敗時回到哪裡
- 這一步如何避免 replay
- 這一步如何觸發撤銷或重試

### 4.3 連線 profile 一旦選定，中途不做劇烈切換

連線建立時可以選擇 profile，但同一條底層連線內不應做大相變。
如果業務型態發生根本改變，應該重新建立新連線或新會話。

### 4.4 伺服器不得持有超出職責的高權限長期秘密

邊緣節點失陷是遲早會發生的事，不是理論風險。
因此線上節點必須是有限權的。

### 4.5 遙測默認最小化

任何遙測、日誌、錯誤回報都可能變成關聯資料。
系統要預設最小化，而不是先收集再談脫敏。

## 5. 提議中的 v3 架構

Chameleon v3 採用五層結構，而不是單純三層。

### 5.1 Local Interface Layer

本地接入層負責與作業系統或應用程式對接，提供一致的入口。

支援模式：

- SOCKS5
- HTTP Proxy
- TUN 模式
- Headless / CLI 模式

這一層只做三件事：

- 接收本地流量
- 做最小必要的分類
- 把流量交給 Session / Forwarding 層

不要在本地接入層塞複雜策略，否則多端會分裂。

### 5.2 Session Core

這是整個系統的安全核心。

它負責：

- 設備身份
- 會話建立
- 授權驗證
- 重放防護
- 金鑰派生
- 會話恢復
- 設備撤銷

關鍵要求：

- 長期身份與短期會話分離
- 每個會話獨立密鑰
- 重新連線不等於重用整個安全上下文
- 會話恢復必須有時間窗與使用限制
- 設備級撤銷必須快於整體重新佈署

### 5.3 Transport Adapter Layer

這一層負責底層承載，但必須被明確降級為「適配器」而不是安全核心。

原則：

- 優先使用成熟、廣泛部署、平台支援好的承載
- 不自行創造新傳輸協議
- 不在 adapter 層做業務邏輯
- 不讓 adapter 決定身份語義

每個 adapter 都需要能力矩陣：

- 支援的平台
- 建連成本
- 多路復用能力
- 切網恢復能力
- 電量成本
- 弱網穩定性
- 可觀測暴露面

只有通過矩陣評估的 adapter 才能進生產。

### 5.4 Forwarding Engine

這是高性能資料面的核心。

它負責：

- 流量多路復用
- 優先級與公平排程
- 回壓控制
- 失敗重試
- DNS 策略
- 路由分流
- 佇列管理

設計重點：

- 不在應用層重造擁塞控制
- 讓 transport 處理 transport 該做的事
- 在 forwarding 層處理業務級排程與回壓
- 使用明確的 stream 優先級，避免大流量壓死互動流

### 5.5 Control Plane

控制面是最容易被低估，也最容易毀掉整個系統的一層。

它負責：

- 節點清單更新
- 功能旗標
- profile 下發
- 信任材料更新
- 版本相容性治理
- 災難回滾
- 最小化遙測

控制面必須做到：

- 可簽名
- 可回滾
- 可審計
- 可分批灰度
- 可離線快取

## 6. 安全模型

### 6.1 身份模型

v3 不應只依賴共享密碼。

更合理的方向是：

- 使用設備級身份
- 長期身份只用於獲取短期授權
- 日常連線盡量使用短期材料
- 支援設備單獨撤銷

這樣做的原因不是理論好看，而是運維可行。
當某台設備遺失、洩露或被懷疑失守時，不能只能靠全網換密碼。

### 6.2 金鑰與信任鏈

v3 應採用分層信任鏈：

- 離線根信任
- 線上簽發層
- 邊緣短期憑據
- 會話級材料

原則：

- 根信任不進線上節點
- 線上節點只拿有限權短期材料
- 會話金鑰頻繁更新
- 控制面更新必須驗簽

### 6.3 重放防護

重放防護不是一句「我們有 nonce」就算完成。

需要明確：

- 哪些請求是可重放的
- 哪些請求必須完整確認後才執行
- 恢復票證的有效時間
- 單票證是否單次使用
- 多設備併發如何處理

v3 的原則是保守處理：

- 默認不讓敏感授權語義進入早期資料
- 把恢復能力視為性能優化，而不是安全基石

### 6.4 節點失陷模型

必須預設邊緣節點會失陷，且在這種情況下保證：

- 歷史內容不會因單節點失陷而被解密
- 不會直接暴露全部設備身份
- 不會導致控制面信任鏈被接管
- 節點可快速撤換

## 7. 性能模型

### 7.1 性能目標不是峰值，而是體感穩定

Chameleon v3 的性能優先級應當是：

1. 弱網可用
2. 互動流低尾延遲
3. 連線恢復快
4. 長連線穩
5. 行動端耗電可接受
6. 之後才是大吞吐

如果這個優先級不明確，系統最終會被下載速度綁架。

### 7.2 擁塞與佇列管理

v3 不在應用層複製 transport CC，而是做三件更有價值的事：

- stream 級排程
- per-class 佇列管理
- 回壓與優先級控制

建議重點觀察：

- 網頁互動流
- 小物件請求
- 長流下載
- DNS / 控制流

這些流不應互相拖死。

### 7.3 Profile 要少、穩、可測量

v3 不應有大量 profile。

建議只保留少數幾種明確用途的 profile，並遵守：

- profile 在建連前選定
- 連線內不做劇烈相變
- profile 有資源預算
- profile 有退出條件
- profile 可被 A/B 測量

如果 profile 變成魔法參數集合，系統一定不可維護。

### 7.4 行動端優先

真正難的是行動端，不是桌面。

因此性能驗證要優先看：

- ARM CPU 開銷
- 背景保活
- Wi-Fi / 蜂窩切換恢復
- 喚醒頻率
- 電量影響

只在桌面測得漂亮數據沒有意義。

## 8. 偽裝與暴露面策略

v3 不再把「偽裝」當成一個獨立魔法模組，而是把它視為
**暴露面管理問題**。

重點不是「我怎麼裝得像別人」，而是：

- 我有沒有額外暴露非必要特徵
- 我有沒有使用不合時宜的參數組合
- 我有沒有在時間上做出異常切換
- 我有沒有在錯誤處理上顯得奇怪
- 我有沒有在部署層暴露出過於集中的資產形態

這種思路更務實，也更容易驗證。

## 9. 服務端與部署策略

### 9.1 不把部署簡化成單容器神話

單容器部署可以存在，但它只能是開發與最小化落地方案，不應被當成
唯一推薦架構。

生產環境至少要考慮：

- 節點分層
- 密鑰分層
- 配置分發
- 遙測回收
- 日誌最小化
- 資產輪換

### 9.2 誘餌站點應是真實服務，不是特效頁

如果需要公開表面，應該部署一個真實服務，而不是只放一個裝飾頁。

原因很簡單：

- 真實服務的行為一致性更高
- 快取、Header、錯誤語義更自然
- 差分探測下更不容易露出人工拼接痕跡

但這個公開表面必須與特權邏輯隔離，不能讓整個站點變成一個脆弱的
混合體。

### 9.3 資產多樣性

如果設計真的面向高壓環境，就不能假設：

- 單一域名足夠
- 單一供應商足夠
- 單一 ASN 足夠
- 單一證書策略足夠

v3 應該從設計上接受資產多樣性，而不是後期被動補洞。

## 10. 多端架構

### 10.1 桌面端

桌面端的核心是穩定代理接入、明確狀態呈現、低維護成本。

要求：

- SOCKS / HTTP 接入一致
- 全局模式與規則模式可切換
- 崩潰後恢復清楚
- 除錯輸出受控

### 10.2 行動端

行動端是首要戰場。

要求：

- 系統 VPN API 接入
- 網路切換恢復
- 背景生命週期管理
- 電量預算
- 小螢幕上的可診斷性

設計上必須承認：

- 行動端的 adapter 選擇更少
- 記憶體更緊
- 背景行為更受限
- 實際體感更敏感

### 10.3 Headless / CLI

這一模式很重要，因為它最適合做：

- 自動化測試
- 弱網模擬
- 大規模回歸
- 節點探活

不要把 CLI 當附帶功能，它應該是驗證與營運工具鏈的一部分。

## 11. 驗證方法

如果沒有嚴格驗證，所有設計都只是自我說服。

v3 建議的驗證分為六類：

### 11.1 表面一致性驗證

檢查：

- 握手參數是否偏離目標棧
- 錯誤回應是否異常
- 生命週期是否存在人工相變
- 多平台之間是否產生可聚類差異

### 11.2 弱網性能驗證

在受控環境中模擬：

- 高延遲
- 高抖動
- 隨機丟包
- 重排序
- 短時斷路

觀察：

- goodput
- p95 / p99 延遲
- 恢復時間
- 失敗重試行為

### 11.3 行動端成本驗證

測：

- CPU
- 記憶體
- 電量
- 背景保活成功率
- 網路切換恢復成功率

### 11.4 安全狀態機驗證

檢查：

- 撤銷是否即時生效
- 票證過期是否可控
- 重放是否能被正確拒絕
- 恢復流程是否會繞過授權邊界

### 11.5 失陷與回滾演練

模擬：

- 邊緣節點失陷
- 配置簽名輪換
- 控制面回滾
- 節點大規模下線

觀察系統是否能在可接受時間內恢復。

### 11.6 多端互通回歸

每次修改必須驗證：

- 桌面到服務端
- 行動端到服務端
- CLI 到服務端
- 舊版本客戶端與新節點
- 新版本客戶端與舊節點

## 12. 最小可行路線

如果要真正開始做，不應同時追求所有特性。

建議路線如下：

### 階段一：內核正確

- 定義身份模型
- 定義會話狀態機
- 定義金鑰與撤銷模型
- 建立最小 forwarding engine
- 建立最小 local interface

### 階段二：性能打底

- 做 stream 排程
- 做佇列管理
- 做弱網恢復
- 做切網恢復
- 做行動端成本優化

### 階段三：表面治理

- 收斂可觀測參數
- 量測 profile
- 做錯誤語義一致性檢查
- 做多平台表面差異檢查

### 階段四：控制面與營運

- 做簽名配置更新
- 做節點輪換
- 做灰度與回滾
- 做最小遙測

這個順序不能反。先追求表面，再補安全內核，最後通常會重寫。

## 13. 最後的判斷

如果用一句話總結 v3 的立場，就是：

**Chameleon 不應該被設計成一個靠技巧賭運氣的代理，而應該被設計成
一個在高干預環境下仍然可控、可度量、可恢復的安全轉發系統。**

這意味著幾件事：

- 接受不存在完美偽裝
- 接受不存在永遠有效的單一承載
- 接受性能必須以穩定性與功耗為約束
- 接受部署與信任鏈比協議包裝更重要
- 接受真正難的是長期演進，不是第一次連通

如果團隊同意這個方向，那麼下一步不應該急著討論更多技巧，而應該
把以下三份正式文檔先寫出來：

1. `threat_model.md`
2. `session_state_machine.md`
3. `control_plane_trust_chain.md`

這三份文檔如果寫不紮實，後面所有的「性能優化」和「表面治理」
都會失去依據。
