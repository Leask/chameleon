# Chameleon 設計審查與重構 v5.0
*(面向通用安全覆蓋網路的完整工程稿)*

## 0. 文檔定位

這份文檔不是 `design_4.md` 的輕量修補版，而是一次正式重構。

我們到目前為止形成的共識已經很清楚：

- 系統不能建立在脆弱技巧上
- 安全必須建立在清晰的狀態機與信任鏈上
- 性能優化必須服從 goodput、恢復能力與功耗
- 多端一致性比單點極致性能更重要
- 控制面與資料面必須分離
- 邊緣節點、設備、配置與更新都應預設可能失陷

這些共識是對的，但 `design_4.md` 仍然帶著明顯的工程過度自信。
它已經不再是概念堆砌，但它仍然把一些尚未驗證的判斷，寫成了像是
可以直接開工的結論。

因此，v5 的目標不是把文檔寫得更像藍圖，而是把它寫得更像一個
真正能長期維護、能多端演進、能經受事故與版本變更的系統規格。

## 1. 內化後的設計目標與思想

### 1.1 我們真正要做的是什麼

Chameleon 應被定義為一個：

- 面向授權場景的安全覆蓋網路
- 可在不穩定與高延遲公網上穩定工作的遠端連通系統
- 具備清晰身份邊界、會話邊界與運維邊界的工程產品
- 可以被桌面端、行動端與無頭模式共同使用的共享內核

它不是一個依靠表面技巧維持生存的「協議花活」。

### 1.2 我們的設計原則

設計應同時滿足以下原則：

- 安全邊界先於傳輸選型
- 身份與授權先於性能優化
- 控制面可靠性先於節點數量
- 多端共享內核先於平台特化
- 可回滾與可恢復先於功能堆疊
- 默認最小權限、最小遙測、最小保存

### 1.3 我們從前幾版得到的知識

我們已經明確學到幾件事：

- 隨機擾動不會自動帶來安全
- 人工相變會製造新特徵，也會讓系統更難驗證
- 不應在應用層重造傳輸層的擁塞控制
- 0-RTT 不是安全內核，只能是小心使用的恢復優化
- 根信任絕不能下沉到邊緣節點
- 多端能力的真正難點在接入模型與生命週期，不在語言選型
- 工程系統最容易失敗的地方通常是控制面、配置治理與回滾

### 1.4 非目標

v5 明確排除以下非目標：

- 不追求不可驗證的「完美隱形」
- 不追求單一節點或單一承載永遠可用
- 不追求峰值吞吐優先於穩定體感
- 不把運維便利性建立在犧牲信任鏈之上
- 不把平台差異假裝成抽象已解決

## 2. 對 `design_4.md` 的嚴厲攻擊

`design_4.md` 最大的問題，不是方向錯，而是把幾個局部正確的思路，
提前包裝成了「已具備代碼架構成熟度」。

這個判斷過頭了。

### 2.1 它把「威脅模型」寫成了偏靜態的願望清單

`design_4.md` 有 threat model，但仍然不夠工程化。

它缺少至少四個東西：

- 能力分級
- 資產分級
- 事件分級
- 失敗後的處置分級

例如：

- 設備私鑰洩露和邊緣節點被入侵，不應進入同一套應對流程
- 控制面配置簽名過期和控制面簽名私鑰疑似洩露，也不應等價處理
- 行動端 session 恢復失敗和線上節點時鐘漂移，也不是同一級事故

`design_4.md` 看起來在定義威脅，實際上仍然停留在「我們知道有哪些風險」
的層面，還沒有變成工程團隊可用的事故分類法。

### 2.2 它把信任鏈講清楚了名詞，卻沒有講清楚操作

離線根、控制面 CA、短期節點憑據，這些詞本身都沒錯。

但 `design_4.md` 沒有回答真正會把系統拖死的問題：

- 誰來執行根信任輪換
- 時鐘漂移如何處理
- 邊緣節點憑據快到期時如何平滑續期
- 控制面與邊緣節點之間如何做雙向驗證
- 控制面被隔離時，客戶端可容忍多久的舊配置
- 配置回滾如何與憑據回滾保持一致

一條信任鏈只有名稱，沒有操作規範，就還不是可運維的信任鏈。

### 2.3 它的狀態機過於樂觀

`design_4.md` 的 Session State Machine 只列了幾個主狀態，這不夠。

真正可落地的狀態機至少要處理：

- 配置過期
- 配置驗簽失敗
- 邊緣節點不可達
- 認證超時
- 時鐘錯誤
- 密鑰輪換失敗
- 會話降級
- 會話遷移失敗
- 設備被撤銷
- 舊版本客戶端不相容

如果狀態機只覆蓋「理想路徑」，那麼代碼最終一定會把錯誤路徑塞進
一堆例外處理與臨時補丁裡，系統會變得不可推理。

### 2.4 它把承載矩陣寫得太簡單

`design_4.md` 的承載矩陣只列了建連成本、多路復用、切網恢復與耗電。
這對真正的產品不夠。

還缺少：

- 平台支援成熟度
- NAT / 防火牆相容性
- 實作複雜度
- 除錯難度
- 庫與供應鏈風險
- 觀測與指標可見性
- 版本升級穩定性

承載不只是網路能力選擇，也是維護成本選擇。

### 2.5 它仍然在讓資料面背負過多語義

`design_4.md` 已經把資料面收斂很多了，但還是有一個問題：
它試圖讓 Forwarding Engine 同時扮演：

- 多路復用器
- 佇列調度器
- 回壓控制器
- DNS 策略中心
- 部分重放防護執行點

這樣做會讓資料面變成新的複雜核心。

資料面應該高效，但也應該薄。當太多策略沉到資料面時：

- 測試會變難
- 跨平台共享會變難
- 性能問題與安全問題會互相糾纏

### 2.6 它沒有真正處理版本治理

一個長期存在的系統，不可能只有一個版本。

`design_4.md` 幾乎沒談：

- 協議版本如何協商
- 不相容變更如何灰度
- 舊節點如何被淘汰
- 舊客戶端如何被限制能力而不是直接崩掉
- 配置 schema 如何演進

如果版本治理不進主設計，未來任何一次升級都可能變成停機事件。

### 2.7 它低估了觀測與隱私之間的衝突

`design_4.md` 有提 Prometheus metrics，也提最小遙測，但沒有定義邊界。

真實系統必須回答：

- 哪些指標可以上報
- 哪些指標只能本地留存
- 指標是否按設備、按節點、按會話聚合
- 事故調試需要的資料是否會反過來變成敏感資料

如果沒有明確分界，最終就會在營運壓力下收集過多資料。

### 2.8 它把「核心庫用 Rust」寫成了答案，而不是選擇

Rust 很可能是合理選擇，但語言不是架構。

如果沒有先定義：

- 跨語言 ABI 邊界
- 記憶體所有權規則
- FFI 錯誤模型
- 行動端綁定策略
- 升級與相容性政策

那麼「用 Rust 寫 core」只是把困難移到了包裝層。

### 2.9 它過早宣稱自己已經可以直接轉成代碼架構

這是 `design_4.md` 最該被攻擊的一句話。

現在的狀態，還不足以直接開工寫完整產品。原因不是想得不夠多，
而是很多關鍵問題還沒有被規格化：

- 狀態機的錯誤分支
- 版本治理
- 控制面離線行為
- 憑據續期規則
- 設備撤銷與恢復
- 觀測分層
- 多端 ABI 穩定性

在這個時點直接開工，最可能的結果是：

- 一個能跑通的 demo
- 一個很快堆滿例外的 core
- 一個越改越像一次性項目的控制面

## 3. v5 的完整設計方向

### 3.1 系統拆分：六層而不是四層

v5 建議採用六層結構：

1. Local Interface Layer
2. Policy Layer
3. Session Core
4. Transport Adapter Layer
5. Forwarding Plane
6. Control Plane

這裡新增 `Policy Layer`，原因很直接：

- 不應讓本地接入層承擔策略
- 不應讓資料面直接承擔所有策略
- 不應讓控制面配置直接散落到各模組內

Policy Layer 的作用是把：

- 路由規則
- 連線分類
- session profile 選擇
- 退避與重試策略
- 本地網路例外規則

集中成一個可測試、可替換、可回滾的邏輯層。

### 3.2 Local Interface Layer

職責非常單純：

- 接收本地流量
- 轉換成統一內部請求
- 維護與作業系統的接入關係

它不應負責：

- 密鑰邏輯
- 路由策略
- profile 判定
- 複雜重試

桌面、行動端、CLI 的差異主要留在這一層。

### 3.3 Policy Layer

這是 v5 新增的關鍵層。

它負責：

- 根據配置與本地狀態決定是否直連或進入覆蓋網路
- 根據請求類型決定 session class
- 根據網路狀態決定使用哪種 transport adapter
- 定義 backoff、failover、冷卻時間
- 向資料面輸出簡單、明確的執行決策

Policy Layer 應該是純邏輯層，儘量避免依賴平台特性。

### 3.4 Session Core

這一層仍然是安全中心，但 v5 進一步要求它明確拆成四個子域：

- Identity Manager
- Session Authenticator
- Rekey Manager
- Revocation Manager

這樣拆的原因是，這四者的錯誤語義不同，測試方式也不同。

#### Session Core 的關鍵規則

- 長期身份只用於獲取短期授權
- 會話金鑰必須可獨立輪換
- 恢復票證不能取代完整授權
- 撤銷必須作用於設備級與會話級
- 時鐘異常必須有降級策略

### 3.5 Transport Adapter Layer

這一層不做安全語義，只提供承載能力。

每個 adapter 都必須實作統一介面：

- `dial()`
- `resume()`
- `close()`
- `health()`
- `capabilities()`

`capabilities()` 至少要返回：

- multiplexing
- migration_support
- estimated_cost
- battery_class
- platform_support
- observability_level

這樣上層才能做能力感知，而不是硬編碼傳輸優先序。

### 3.6 Forwarding Plane

v5 不再稱它為 engine，而稱為 plane，因為它應該是一個資料平面，
不是第二個控制中心。

它的責任限定為：

- 流量封裝與解封裝
- stream 映射
- 回壓
- 佇列優先級
- 輕量重試
- 速率觀測

它不應直接承擔：

- 大部分路由決策
- 設備授權判定
- 配置治理

### 3.7 Control Plane

控制面必須由四個明確組件構成：

- Config Service
- Credential Service
- Rollout Service
- Telemetry Service

每個組件各自有邊界：

- Config Service 只管配置內容與版本
- Credential Service 只管簽發、續期、撤銷
- Rollout Service 只管灰度、凍結、回滾
- Telemetry Service 只管受限可觀測資料

## 4. v5 的狀態機設計

### 4.1 設備生命週期狀態

設備應至少包含以下狀態：

- `DEVICE_UNREGISTERED`
- `DEVICE_REGISTERING`
- `DEVICE_ACTIVE`
- `DEVICE_SUSPENDED`
- `DEVICE_REVOKED`
- `DEVICE_RECOVERING`

這些狀態不能只存在控制面，也必須在客戶端本地有清晰映射。

### 4.2 會話狀態機

v5 建議至少包含以下狀態：

- `SESSION_IDLE`
- `SESSION_DISCOVERING`
- `SESSION_CONNECTING`
- `SESSION_AUTHENTICATING`
- `SESSION_ESTABLISHED`
- `SESSION_DEGRADED`
- `SESSION_REKEYING`
- `SESSION_MIGRATING`
- `SESSION_DRAINING`
- `SESSION_CLOSED`
- `SESSION_REVOKED`

#### 補充要求

- 每個狀態都要定義可進入事件與可退出事件
- 每個失敗都要映射到可觀測錯誤碼
- 每個錯誤碼都要有是否可重試的標記
- 每個重試都要服從 Policy Layer 下發的退避規則

### 4.3 配置狀態機

客戶端配置不應被視為靜態文件，而應是一個有狀態的對象：

- `CONFIG_EMPTY`
- `CONFIG_STAGED`
- `CONFIG_ACTIVE`
- `CONFIG_EXPIRED`
- `CONFIG_ROLLBACK`
- `CONFIG_REJECTED`

這樣才能合理處理：

- 簽名合法但內容不相容
- 新配置下載成功但連通性下降
- 控制面暫時不可達

## 5. 安全模型與信任鏈

### 5.1 分層信任鏈

v5 沿用分層信任，但補足操作規範：

- 離線根只負責簽發上層信任與緊急輪換
- 線上簽發層有獨立審計與最小權限
- 邊緣節點只持有限時、限用途材料
- 客戶端本地只保存執行所需的最小材料

### 5.2 憑據續期

憑據續期必須定義：

- 續期開始窗口
- 續期失敗重試規則
- 續期期間是否允許舊憑據繼續工作
- 時鐘不可信時的行為
- 緊急撤銷時的優先順序

### 5.3 撤銷設計

撤銷至少分三類：

- 設備撤銷
- 邊緣節點撤銷
- 簽發層材料撤銷

三者不能混成一種「黑名單」機制。

### 5.4 0-RTT 與恢復

v5 對這件事採取非常保守的規則：

- 恢復是性能優化，不是基本能力
- 可恢復不代表可跳過完整授權
- 任何帶副作用的語義都不應依賴早期資料
- 恢復流程失敗必須快速回落到完整會話建立

## 6. 性能模型

### 6.1 產品級 SLO

沒有 SLO 的性能討論沒有意義。

v5 應定義至少以下目標：

- 首次可用連線時間
- 斷線後恢復時間
- p95 互動請求延遲
- 長連線穩定性
- 行動端每小時額外耗電預算
- 設備常駐記憶體上限

### 6.2 資料面排程

資料面建議分四類流量：

- control
- interactive
- bulk
- background

每一類都要有：

- 優先級
- 佇列上限
- 丟棄策略
- 回壓策略

### 6.3 不把表面治理寫進核心性能路徑

任何會引入顯著 CPU、記憶體、耗電開銷的表面治理手段，都不應直接
寫進核心性能路徑。

先把：

- 握手時間
- 連線恢復
- 排程公平性
- 弱網 goodput
- 行動端功耗

做紮實，再談其他優化。

## 7. 觀測、隱私與事故處置

### 7.1 遙測分層

v5 建議把遙測分成三層：

- 本地除錯資料
- 匿名聚合指標
- 事故級臨時診斷資料

三者必須明確分離，不能混用。

### 7.2 預設最小化

默認狀態下，系統只應保留：

- 健康狀態
- 錯誤碼計數
- 大類型性能指標

不應默認保存：

- 精細會話追蹤
- 目的地細節
- 長期設備行為序列

### 7.3 事故處置

事故至少要分成：

- 連通性事故
- 簽名與憑據事故
- 客戶端版本事故
- 控制面事故
- 節點容量事故

每一類都應有明確 runbook。

## 8. 多端實作策略

### 8.1 Core 的條件

如果使用 Rust 作為 core，必須先定義：

- 穩定 ABI
- 錯誤碼表
- 記憶體所有權規則
- 非同步回調模型
- 版本相容性政策

否則跨平台封裝層會變成新的不一致來源。

### 8.2 桌面端

桌面端應優先確保：

- 接入簡單
- 狀態明確
- 崩潰可恢復
- 日誌可控

### 8.3 行動端

行動端應優先確保：

- 系統 VPN API 的穩定整合
- 切網恢復
- 背景限制下的可靠性
- 電量與記憶體預算

### 8.4 CLI / Headless

CLI 不只是附屬模式，而是：

- 自動化測試入口
- 節點驗證工具
- 容量測試入口
- 回歸測試入口

## 9. 驗證路線

### 9.1 先驗證狀態機，再驗證性能

不要反過來。

應優先做：

- 狀態轉換單元測試
- 撤銷與恢復測試
- 配置回滾測試
- 時鐘漂移測試
- 版本不相容測試

### 9.2 再做性能與弱網

之後才做：

- 高延遲
- 抖動
- 丟包
- 重排序
- 切網
- 節點失效

### 9.3 最後做多端一致性

最後要驗證：

- 桌面與行動端狀態碼一致
- CLI 與 GUI 對同一錯誤有一致語義
- 同一配置在不同平台結果一致

## 10. 最小可行開發順序

v5 建議的開發順序如下：

1. 先寫 `threat_model.md`
2. 再寫 `state_machine.md`
3. 再寫 `trust_chain_ops.md`
4. 之後寫 `policy_schema.md`
5. 再開始最小 core
6. 再做單一 adapter
7. 再做單一平台接入
8. 最後才擴多平台與多 adapter

如果順序反過來，團隊很快就會陷入：

- 平台修 bug
- 控制面補洞
- 狀態機失控
- 配置 schema 不斷破壞相容性

## 11. 結論

`design_4.md` 的價值在於，它終於把問題拉回了工程世界。
但它最大的缺點也來自這裡：它過早相信自己已經到了可以全面施工的
階段。

v5 的核心結論是：

- 現在還不能宣稱設計完備
- 但已經可以明確知道該把哪些不成熟的部分拆掉
- 下一步不是堆更多功能，而是把狀態機、信任鏈操作與版本治理寫實

如果要用一句話概括 v5：

**真正成熟的安全覆蓋網路，不是由某個巧妙傳輸技巧定義的，而是由
它在失敗、升級、撤銷、切換、事故與恢復中的可預測性定義的。**
