變色龍 (Chameleon) 協議設計與架構白皮書 (Draft v0.1)
1. 摘要與設計哲學
隨著深度封包檢測 (DPI) 與機器學習技術的普及，傳統「基於未知加密特徵」的代理協議（如早期的 Shadowsocks 或自訂 TCP/UDP 協議）已無法在現代嚴苛的網路審查環境（如 GFW）中長期生存。

Chameleon 協議的設計哲學是：「大隱隱於市，絕對寄生」。
本協議放棄創造無法識別的加密流量，轉而將所有代理行為 100% 偽裝成現代互聯網最常見的骨幹標準——HTTP/3 與 WebTransport 流量。透過動態流量整形與極致的應用層擁塞控制，在保證「完美偽裝」的前提下，最大化跨國惡劣網路環境下的傳輸效率。

2. 核心架構總覽
Chameleon 採用 Client-Server 架構，完全架構於 UDP 之上。

底層傳輸 (Transport)： QUIC (HTTP/3)

會話層 (Session)： WebTransport (提供多路復用與雙向 Stream)

應用層 (Application)： 動態流量整形器 (Traffic Shaper) + 自訂擁塞控制 + 代理負載 (Payload)

3. 關鍵設計特點與技術細節
3.1 傳輸層：擁抱 WebTransport 與 QUIC
零隊頭阻塞 (No Head-of-Line Blocking)： 利用 QUIC 原生的多路復用特性，將使用者的每個網路請求對映到獨立的 QUIC Stream。跨國線路中單一封包的遺失，絕對不會卡住其他併發的網頁資源載入。

0-RTT 極速連線： 客戶端會快取伺服器的傳輸參數與加密金鑰，在後續連線時實作 0-RTT 握手，首個 UDP 封包即可攜帶代理請求，消除傳統 TLS/TCP 帶來的 2~3 個 RTT 延遲。

完美融入白名單流量： 伺服器端對外呈現為一個標準的 HTTP/3 網頁伺服器，流量特徵與 Cloudflare、Google 等大廠的現代 CDN 節點完全一致。

3.2 智能流量整形 (Smart Traffic Shaping)
為了對抗基於 AI 的流行為分析（Traffic Profiling），協議內建動態干擾機制。

動態填充 (Dynamic Padding)： 演算法會隨機將有效載荷切碎，或在封包中塞入無意義的隨機字節，徹底改變真實應用的封包大小分佈（Packet Size Distribution）。

時序抖動 (Jitter Injection)： 刻意微調封包發送的時間戳，將穩定、連續的長連接（如 4K 影片串流）打散，使其在統計學上模擬成「使用者正在頻繁點擊網頁」的突發性流量 (Bursty Traffic)。

3.3 效能與偽裝的動態平衡 (階段性卸載)
為解決「高強度偽裝導致網速下降」的矛盾，Chameleon 實作了狀態感知的傳輸策略：

審查高危險期（連線前 5-10 秒）： 審查系統的機器學習模型通常在此階段收集特徵。此時 Chameleon 會啟動最高級別的 Padding 與 Jitter，寧可犧牲部分帶寬，也要確保連線被審查系統判定為「合法網頁瀏覽」。

穩定傳輸期（大流量爆發）： 當判定已度過審查系統的特徵收集視窗後，協議會動態卸載偽裝機制。將 Padding 比例降至 5% 以下，消除 Jitter，將頻寬 100% 讓渡給真實的有效載荷。

3.4 應用層擁塞控制 (Aggressive Congestion Control)
無視系統限制： 由於運行在 UDP 之上，Chameleon 實作了應用層的 BBRv3（或類似 Brutal 的激進算法）。

頻寬搶佔： 在面對 GFW 的隨機丟包或 QoS 限速時，不採取傳統 TCP 的「退讓減速」策略，而是基於頻寬探測強制維持發送速率，確保在極端惡劣的跨國線路上依然能跑滿頻寬。

3.5 認證機制與絕對靜默 (Silent Drop & Anti-Probing)
隱藏密鑰交換： 不存在傳統的代理登入指令。認證資訊被加密雜湊後，隱藏於 TLS 1.3 Client Hello 的擴充欄位（Extension）或特定的 HTTP/3 Header 中。

真理之門 (主動探測防護)： 若防火牆發送主動探測封包（未攜帶正確隱藏密鑰的常規訪問），伺服器不會切斷連線或報錯，而是無縫將請求反向代理至一個真實的、高流量的合法網站（如維基百科或開源社群鏡像站）。探測器得到的永遠是無懈可擊的正常網頁回覆。

4. 部署與營運考量
極簡容器化： 系統服務應打包為單一 Docker Image。透過 docker-compose，管理者只需填入網域與密碼，系統內部會自動處理 QUIC 監聽、Let's Encrypt 憑證的自動申請與續期。

多端支援： 由於依賴標準化的 Web API，客戶端可以輕易使用 Node.js、Go，甚至在不具備底層網路權限的環境中運行。

5. 待解決的工程挑戰
CPU 運算開銷： 在行動裝置上，頻繁的封包 Padding 計算與應用層的擁塞控制可能會增加設備耗電量。需要針對 ARM 架構進行演算法的最佳化。

動態卸載的觸發條件： 如何精準判定「審查視窗已過」，目前仍需依賴對特定防火牆行為的逆向工程與經驗參數，這部分需要設計成可雲端動態下發更新的設定檔。
