# Chameleon 設計審查與修正方向（design_1）

## 1. 文檔定位

這份文檔不是對原方案的補強版實作指南，而是對 `design.md`
的架構性反駁、風險澄清與重構建議。

目標是避免在錯誤前提上投入工程資源，先把威脅模型、安全邊界、
性能路徑與多端可落地性定義清楚，再決定是否值得進入 PoC。

本文刻意不提供具體的對抗性繞過細節、探測對抗技巧或規避策略。
重點是把系統設計做得更嚴謹、更可驗證、更容易演進。

## 2. 對原始設計的核心判斷

原設計最嚴重的問題，不是某一個協議選型錯誤，而是整體上把
「使用常見協議」誤當成「不可識別」，把「行為擾動」誤當成
「更安全」，把「激進傳輸」誤當成「更高效」。

這三個前提如果不改，後續不論怎麼調參，系統都很容易出現以下
結果：

- 看起來像常見流量，但細節指紋明顯異常
- 想靠擾動隱身，最後反而造出新的穩定特徵
- 想在惡劣網路中硬頂帶寬，最後得到更差的 goodput
- 客戶端與服務端的實作變得複雜，但可驗證性下降
- 一旦外部環境改變，整個系統沒有清晰的調整框架

因此，這份設計目前更像是概念堆疊，而不是可落地的系統規格。

## 3. 高優先級問題

### 3.1 缺失威脅模型

原文沒有明確定義：

- 對手能看到什麼
- 對手能主動做什麼
- 對手是即時封鎖、延遲封鎖，還是關聯分析
- 服務端失陷後的後果
- 使用者設備被觀察時的風險
- 可接受的失敗模式是什麼

沒有威脅模型，下面所有「偽裝」「效率」「安全」的說法都無法
驗證，也無法做取捨。

### 3.2 將 HTTP/3 / WebTransport 等同於完美偽裝

這是原方案最危險的假設。

流量是否會被識別，不只取決於協議名義上是不是 HTTP/3 或
WebTransport，還取決於整個行為表面是否一致，包括但不限於：

- TLS / QUIC 指紋
- 版本協商與握手行為
- Header 組成與排序
- 連線生命週期
- stream 使用方式
- 重傳、pacing、ACK 節奏
- 錯誤處理
- IP / ASN / 證書鏈 / 部署位置

也就是說，協議名相同，不代表行為相同。只要底層實作、自訂
擁塞控制、padding、jitter 或連線模式與主流棧不同，就可能比
一個普通系統更容易被聚類。

### 3.3 把流量整形視為主要防線

原文把 dynamic padding 與 jitter injection 放在很核心的位置。
這個方向有兩個問題：

- 擾動會增加成本，卻不保證降低可識別性
- 如果擾動模式固定或半固定，反而會成為新特徵

特別是「把穩定串流打散成突發網頁點擊」這種想法，在工程上很
容易得到一種既不像真實互動網頁、也不像真實串流的中間態。
這種中間態未必更安全，卻幾乎一定更低效。

### 3.4 階段性卸載會引入相變特徵

原文假設前 5 到 10 秒是高危期，之後可以逐步卸載偽裝。

這個假設本身沒有被證明，並且會導致一個更大的問題：流量在
固定時間窗出現明顯相變。只要封包大小分佈、發送節奏、吞吐
占比在某個時間點集體變化，這個變化本身就可能是可識別信號。

如果系統需要 profile，應該避免顯式的「先重、後輕」切換模型。
行為族應當穩定、連續、可配置，而不是基於脆弱的單一時間窗。

### 3.5 自訂激進擁塞控制的收益被高估

原文把「應用層 BBRv3 或類似激進算法」視為高性能來源，這個判
斷過於樂觀。

問題在於：

- QUIC 已經帶有成熟的傳輸控制邏輯
- 在應用層再強行維持發送速率，很可能和傳輸層互相干擾
- 在隨機丟包、抖動、限速下，硬頂速率通常會增加無效重傳與排隊
- 最終使用者真正關心的是 goodput、穩定延遲與耗電，而不是理論
  上的發送速率

因此，傳輸層第一原則應該是穩定與可預測，不是搶帶寬。

### 3.6 認證方案不嚴謹，且可能破壞協議表面

把認證資訊藏進 `ClientHello` 擴展或某些 HTTP/3 Header，不是一個
完整的安全設計，只是一個位置選擇。

原文沒有回答以下問題：

- 重放保護怎麼做
- 前向保密如何保證
- 客戶端憑證撤銷怎麼做
- 金鑰輪換策略是什麼
- 設備遺失後如何失效
- 多設備授權如何管理
- 0-RTT 資料是否會帶來 replay 風險

更重要的是，修改握手可見表面往往就是在創造指紋。

### 3.7 主動探測防護思路過度簡化

原文提出「不合法請求就反代到真實網站」的想法。這不是一個可靠
的保護策略。

原因很直接：

- 真實站點的內容、Header、快取與錯誤語義很難對齊
- 對手可以用畸形請求、邊界條件、並發 stream 與差分探測做比對
- 第三方站點本身會變化，導致鏡像一致性更難維持
- 這會引入法律、濫用、資源耗盡與運維成本風險

「看起來像正常網站」不是一句話就能成立的屬性，而是完整站點
生命週期與部署細節共同產生的外觀。

### 3.8 多端與部署敘述過於樂觀

原文對多端支援與容器化部署的預期偏理想化。

多端真正的難點不在於能不能用標準 API 發包，而在於：

- 桌面系統代理接管方式
- 移動端後台保活與網路切換恢復
- 電量控制
- 斷線重連
- 可觀測性
- 更新信任鏈
- 用戶配置錯誤時的恢復能力

同理，部署也不只是 `docker-compose`、域名與憑證自動化。真正困難
的是密鑰管理、資產隔離、更新策略、風險控制與封鎖後恢復。

## 4. 我對重構方向的建議

### 4.1 先改目標，不要追求「完美模仿」

不應把目標定義為「完整模仿某個熱門協議棧」。

更合理的目標是：

- 內核簡單
- 安全邊界清楚
- 傳輸外層可替換
- 行為配置可測量
- 性能可穩定優化
- 多端接入面一致

這樣系統才有演進空間，不會把所有風險壓在某一種表面包裝上。

### 4.2 用分層架構替代單體協議幻想

建議把系統拆成三層：

1. `transport adapter`
2. `secure session`
3. `forwarding engine`

其中：

- `transport adapter` 只負責承載與連通性抽象
- `secure session` 負責身份、會話建立、重放防護與密鑰更新
- `forwarding engine` 負責多路復用、流量分類、重試與策略控制

這個拆法的好處是，外層承載可以替換，但安全語義與轉發邏輯不需
要跟著重寫。

### 4.3 安全設計應明確，不依賴「藏在哪裡」

新的設計應先回答安全問題，再討論性能優化。

至少要明確規定：

- 身份模型
- 會話建立流程
- 單次會話與長期身份的關係
- 重放保護機制
- 金鑰輪換機制
- 設備撤銷流程
- 配置更新的信任來源

認證不是「把資訊藏進某個看不見的位置」，而是要形成可審計的
協議狀態機。

### 4.4 性能優化以 goodput 與穩定性為核心

性能設計不應優先追求理論上的滿速，而應優先追求：

- 穩定 goodput
- 低尾延遲
- 低功耗
- 可預測重連
- 在弱網與切網場景下恢復快

如果一個優化能提高峰值吞吐，但明顯惡化耗電、重傳或尾延遲，那
它未必是對使用者有價值的優化。

### 4.5 流量策略應該「少而穩」，不要「多而怪」

如果系統需要流量 profile，應當遵守兩個原則：

- profile 數量少
- 單個 profile 長期穩定

不要依賴大量隨機擾動、固定時間窗切換或大幅相變。因為這些做法
很容易得到統計上異常、工程上昂貴、性能上不穩的結果。

### 4.6 多端從接入抽象開始，而不是從傳輸 API 開始

真正可用的多端方案，應先定義：

- 本地代理介面
- 會話管理 API
- 配置同步格式
- 日誌與可觀測性規格
- 斷線重試與網路切換狀態機

然後再分別實作：

- 桌面代理接入
- 行動端系統整合
- CLI / headless 模式

如果這些抽象不先定義，單純說「Node.js、Go、Web API 都能做」
沒有實際意義。

## 5. 建議的新文檔骨架

如果要把原始方案重寫成一份能進入工程設計的規格，我建議新的
主文檔至少包含以下章節：

### 5.1 系統目標

- 要解決什麼問題
- 不解決什麼問題
- 性能目標是什麼
- 安全目標是什麼

### 5.2 威脅模型

- 觀察者能力
- 主動干預能力
- 封鎖與探測能力
- 資產失陷模型
- 用戶風險邊界

### 5.3 架構分層

- transport adapter
- secure session
- forwarding engine
- client local interface
- control plane

### 5.4 會話與身份

- 身份註冊
- 授權與撤銷
- 金鑰輪換
- 重放防護
- 配置更新信任鏈

### 5.5 傳輸與性能

- 基線傳輸棧
- 擁塞控制策略
- 流量 profile 模型
- 重試與恢復策略
- 弱網與切網測試方法

### 5.6 客戶端架構

- 桌面接入
- 行動端接入
- 無頭模式
- 配置同步
- 遙測與除錯

### 5.7 服務端與營運

- 部署拓撲
- 日誌與可觀測性
- 事故處置
- 更新策略
- 密鑰與資產管理

### 5.8 驗證計畫

- 單元測試
- 協議互通測試
- 弱網測試
- 資源使用測試
- 行為一致性測試

## 6. 我認為應該立刻補做的工作

在任何 PoC 之前，先做以下事情：

1. 寫出正式威脅模型
2. 寫出會話安全模型
3. 定義可接受失敗模式
4. 定義性能基線與測量方法
5. 定義多端接入抽象
6. 把控制面與資料面分開

如果這六件事不先完成，PoC 很可能只是在驗證某個暫時可跑通的
技巧，而不是在驗證一個能長期維護、能多端演進的系統。

## 7. 結論

原設計最大的問題，不是野心太大，而是把一些沒有被證明的假設
直接當成架構前提：

- 常見協議外觀等於不可識別
- 擾動越多越安全
- 帶寬搶得越兇越高效
- 主動探測可以靠反代真站掩護

這些前提都不穩。

下一版設計應該把重心從「如何看起來像某種東西」移到
「如何定義清楚的安全與性能內核」，並採用可替換外層承載、
明確會話安全、穩定性能優化和可落地多端抽象的架構方向。

只有這樣，後續的工程投入才有積累價值。
